# Flutter/Dart Support for Claude Context — Design Spec

**Date:** 2026-04-02  
**Status:** Draft  
**Scope:** `packages/core` + `packages/vscode-extension`

---

## Problem Statement

The current implementation is not optimized for Flutter developers. Key gaps:

1. **No Dart AST support** — `.dart` files fall back to LangChain character splitter, which splits by character count without understanding class/method boundaries. This degrades semantic search quality significantly for Flutter codebases.
2. **No default Dart file indexing** — `.dart` files must be manually added via `customExtensions`, adding friction for every new user.
3. **No Flutter ignore presets** — Auto-generated files (`*.g.dart`, `*.freezed.dart`, `*.gr.dart`, `*.gen.dart`) produced by `build_runner` are indexed by default, polluting the vector database with noise that degrades search relevance.
4. **No Flutter project auto-detection** — The extension has no awareness of whether the workspace is a Flutter project, requiring manual configuration.

---

## Chosen Approach: Full-Stack Flutter Support (Option C)

Fix all four issues across both `core` and `vscode-extension`, so all claude-context consumers (VS Code, Chrome, MCP) benefit from the core changes.

## Architecture

### Layer 1: `packages/core` — Dart AST Splitting

**File:** `src/splitter/ast-splitter.ts`

Add `tree-sitter-dart` as a dependency and register Dart in the AST language map.

**Dart-specific node types to split on:**
```
class_definition
function_signature
method_signature
constructor_signature
getter_signature
setter_signature
mixin_declaration
extension_declaration
```

These map to the logical code units in Dart that correspond to what a developer searches for. `function_signature` covers top-level functions; `method_signature` covers class methods; `constructor_signature` covers factory constructors and named constructors; `getter_signature`/`setter_signature` cover computed properties.

**Language map entry:**
```typescript
'dart': { parser: Dart, nodeTypes: SPLITTABLE_NODE_TYPES.dart },
```

No fallback needed — tree-sitter-dart is mature and handles all valid Dart syntax.

### Layer 2: `packages/core` — Default Extensions & Ignore Patterns

**File:** `src/context.ts` (specifically `DEFAULT_SUPPORTED_EXTENSIONS` and `DEFAULT_IGNORE_PATTERNS` constants)

#### Default file extensions
Add `.dart` alongside other supported extensions so Flutter projects are indexed without requiring any manual `customExtensions` configuration.

#### Default Flutter ignore patterns
Add to the core default ignore list:

| Pattern | Reason |
|---------|--------|
| `*.g.dart` | Generated by `json_serializable`, `build_runner` |
| `*.freezed.dart` | Generated by `freezed` package |
| `*.gr.dart` | Generated by `auto_route` |
| `*.gen.dart` | Generated by `flutter_gen` |
| `.fvm/` | Flutter Version Manager local SDK cache |

Note: `build/**` is **already** in `DEFAULT_IGNORE_PATTERNS` so it covers Flutter build output. `.dart_tool/` is already excluded by the hidden-directory rule (`pathParts.some(part => part.startsWith('.'))`). No changes needed for those two.

These patterns must be added to the default ignore list (not as hidden-dir heuristic), so they appear in the ignore list regardless of whether the user keeps the default behavior.

### Layer 3: `packages/vscode-extension` — Flutter Auto-Detection

**File:** `src/extension.ts`

On extension activation, run `detectFlutterProject()`:
1. Look for `pubspec.yaml` in each workspace root
2. Read the file content and check for `flutter:` under the `dependencies` or `dev_dependencies` section, OR for the `flutter:` top-level key in pubspec
3. If detected, show a one-time notification (keyed on workspace path in `context.globalState` to avoid repeated prompts):

> "Flutter project detected. Apply Flutter-optimized settings? (Adds `.dart` to indexed extensions and excludes generated files)"

With buttons: **Apply** | **Don't Ask Again**

**On Apply:**
- Merge `['dart']` into existing `customExtensions` (dedup)
- Merge Flutter ignore patterns into existing `customIgnorePatterns` (dedup)
- Save via `configManager.saveIndexingConfig()`
- Show info: "Flutter settings applied. Re-index to take effect."

**On "Don't Ask Again":**
- Store flag in `context.globalState` for this workspace, never prompt again

**New function signature:**
```typescript
async function detectFlutterProject(
  context: vscode.ExtensionContext,
  configManager: ConfigManager
): Promise<void>
```

## Data Flow

```
User opens Flutter workspace
        │
        ▼
detectFlutterProject() — checks pubspec.yaml
        │
        ├── Not Flutter ──▶ no-op
        │
        └── Is Flutter ──▶ prompt (once per workspace)
                │
                └── User clicks Apply
                        │
                        ▼
              saveIndexingConfig(mergedExtensions, mergedIgnorePatterns)
                        │
                        ▼
              User runs "Index Codebase"
                        │
                        ▼
              FileSynchronizer (core) honors merged patterns
                        │
                        ▼
              AstCodeSplitter splits .dart files via tree-sitter-dart
                        │
                        ▼
              Milvus receives clean, well-structured Dart chunks
```

## Error Handling

- **`pubspec.yaml` is present but malformed YAML:** Catch parse error, treat workspace as non-Flutter (safe fallback).
- **`tree-sitter-dart` parse error:** Already handled by existing AST splitter fallback to LangChain.
- **User does not re-index after applying settings:** New settings take effect on next sync/index. No automatic re-index triggered (avoids unexpected long operations).

## Testing

### Core (unit tests in `packages/core`)

1. **Dart AST splitting:** Feed a real Dart file with a class, multiple methods, a mixin, and a factory constructor. Assert the output chunks map correctly to each logical unit and no chunk spans two unrelated methods.
2. **Flutter ignore patterns:** Create a mock file tree including `lib/models/user.g.dart`, `lib/models/user.freezed.dart`, `build/app.apk`. Assert these are excluded from `generateFileHashes()`.
3. **`.dart` default extension:** Assert that a file `lib/main.dart` is included in a default (no-config) synchronization.

### VS Code Extension (integration)

1. **Auto-detection positive case:** Workspace with `pubspec.yaml` containing `flutter:` → notification appears.
2. **Auto-detection negative case:** Workspace with `pubspec.yaml` but no `flutter:` key (pure Dart CLI app) → no notification.
3. **Apply settings:** Click Apply → `configManager.getCustomExtensions()` contains `'dart'`, `configManager.getCustomIgnorePatterns()` contains `'*.g.dart'`.
4. **Don't Ask Again:** After dismissal, re-activating extension does not show notification.
5. **Merge, don't overwrite:** User already has `customExtensions: ['swift']` → after Apply, result is `['swift', 'dart']`.

## Out of Scope

- Dart-specific embedding model tuning (orthogonal concern)
- `.arb` (localization) file indexing — not code, different semantic meaning
- `pubspec.lock` indexing — auto-generated, noise
- Flutter web (`web/`) or native (`android/`, `ios/`) platform code — these are handled by existing JS/Java/Kotlin/Swift/ObjC support already in the AST splitter
